## Findings Confirmed By Repo Audit

### Backend

`resolveWebAppBaseUrl()` is the central backend resolver for web-facing absolute URLs. It prefers `WEB_APP_URL`, then `FRONTEND_URL`, then a local fallback built from `WEB_APP_USE_HTTPS`, `WEB_APP_HOST`, and `WEB_APP_PORT`. Non-local markers `qa`, `uat`, `staging`, `production`, and `prod` throw if no web URL is configured.

Backend `.env.example` now defaults local web links to `http://localhost:3000`, documents LAN IP use for device testing, and defines `FRONTEND_PUBLIC_CHECKOUT_CALLBACK_URL=http://localhost:3000/bag/payment-return`.

Email verification links are generated by `EmailVerificationHelperService.generateVerificationLink()` as:

```txt
<web-app-base-url>/verify-email?token=<token>[&next=<path>]
```

The helper only accepts `next` values that start with `/` and rejects protocol-relative values starting with `//`.

Signup and resend-verification both use the same verification-link helper. Brand users receive a post-verification `next` of `/profile?modal=brand-setup&modalOrigin=prompt`; regular users receive `/profile`.

User password reset links are generated as:

```txt
<web-app-base-url>/reset-password?token=<token>
```

Admin password reset links are generated as:

```txt
<web-app-base-url>/admin/reset-password?token=<token>
```

Email-change confirmation links are generated as:

```txt
<web-app-base-url>/settings?tab=account-security&emailChangeToken=<token>
```

Admin account creation emails use:

```txt
<web-app-base-url>/admin/login
```

Brand staff invites use `resolveAppUrl()` and route to:

```txt
<web-app-base-url>/brand/staff/invite?token=<token>
```

Email branding exposes `resolveAppUrl(path)` and currently uses it for studio verification, account reactivation, account-security review, store payout links, and other email CTAs. These all resolve against the same backend web-app base URL.

Payment callback routing is server-owned. `PaymentService.resolveCallbackBaseUrl()` intentionally ignores caller-supplied callback URLs and uses `FRONTEND_PUBLIC_CHECKOUT_CALLBACK_URL` or `<web-app-base-url>/bag/payment-return`. Production startup validates that the checkout callback URL is HTTPS and not loopback/private.

Custom-order checkout resume links use:

```txt
<web-app-base-url>/custom-orders/resume/<token>
```

Notification email routing sanitizes `targetUrl` to internal paths first, then converts allowed internal paths to absolute web URLs before rendering notification email CTAs. Push notification routing keeps internal paths.

Brand public profile/QR/share links use the backend web-app base URL and route to `/u/:username` when a username exists, otherwise `/profile/:ownerId`. Non-local environments reject loopback web hosts for those public links.

### Frontend

`src/config/env.ts` requires `VITE_API_BASE_URL`, exposes optional `VITE_APP_URL`, and configures Axios with `withCredentials` through `VITE_API_WITH_CREDENTIALS`.

`src/api/httpClient.ts` uses `env.api.defaultConfig` for API base URL and credentials. Access tokens are held in volatile memory and attached to API requests. Refresh is performed through `/auth/refresh`.

`src/api/AuthApi.ts` maps frontend auth screens to backend endpoints:
- `POST /auth/password-reset/request`
- `POST /auth/password-reset/confirm`
- `POST /auth/change-password`
- `POST /auth/change-email/request`
- `POST /auth/change-email/confirm`
- session/device/account deletion endpoints
- admin password reset endpoints

`src/App.tsx` registers the web routes that receive backend auth links:
- `/verify-email`
- `/forgot-password`
- `/reset-password`
- `/admin/reset-password`
- `/brand/staff/invite`
- `/settings` with `tab=account-security`
- `/bag/payment-return`
- `/custom-orders/resume/:token`

`ForgotPasswordPage` only requests a reset link. It does not handle a token.

`ResetPasswordPage` reads `token` from the query string, posts it to `/auth/password-reset/confirm`, and removes the token from browser history after success with `window.history.replaceState({}, document.title, '/reset-password')`.

`EmailVerify` reads `token`, `pending`, `email`, and `next` query params. It calls `GET /auth/verify-email` with the token, refreshes `/auth/profile` on success, and sanitizes `next` to same-app relative paths only. It also converts `/studio/store` continuations back to `/profile`.

`AccountSecuritySettings` reads `emailChangeToken` from `/settings?tab=account-security`, calls `/auth/change-email/confirm`, and removes `emailChangeToken` from the URL after completion.

`SettingsHome` maps `account`, `security`, and `account-security` to `AccountSecuritySettings`.

`BrandStaffInvitePage` reads `token`, requires sign-in with the invited email, preserves the invite URL through `/login?next=<encoded current path>`, and then calls accept/reject invite endpoints.

`publicUrlBuilder` uses `VITE_APP_URL` for frontend-generated public absolute URLs, with `window.location.origin` as fallback. This is separate from backend-generated email auth links.

Checkout currently passes `window.location.origin + '/bag/payment-return'` to the backend, but the backend intentionally ignores client-provided callback URLs and resolves the authoritative callback from server environment.

### Mobile

`app.json` defines the Expo scheme `threadlymobile`. It does not define iOS associated domains or Android app-link intent filters for universal/app links.

`src/config/env.ts` exposes:
- `EXPO_PUBLIC_API_BASE_URL`
- `EXPO_PUBLIC_WEB_APP_URL`
- `EXPO_PUBLIC_TRUSTED_WEB_ORIGINS`
- API credential/storage keys

The mobile default `EXPO_PUBLIC_WEB_APP_URL` fallback is `http://localhost:5173`, while `.env.example` instructs device testing to use a LAN IP and port `3000`.

`src/api/AuthApi.ts` has both `requestPasswordReset(email)` and `confirmPasswordReset(token, newPassword)`, but the audited app routes do not include a reset-password screen.

`app/(auth)/_layout.tsx` registers only `login`, `signup`, and `forgot-password` under the auth stack.

`app/(auth)/forgot-password.tsx` requests a password reset email and then tells the user to follow the link in the email. It does not open or handle the reset token.

`app/_layout.tsx` installs Expo Linking listeners for initial URLs and foreground URL events, then delegates to `handleDeepLink()`.

`src/utils/notificationRouting.ts` only handles deep links for message/inbox URLs. It does not route `/reset-password`, `/verify-email`, `/settings?tab=account-security`, or `/brand/staff/invite`.

Mobile Studio WebView routing uses `env.webAppUrl` and `env.trustedWebOrigins` for trusted web origins and Studio paths. That web URL is not currently used for native auth-link handling.

No mobile `reset-password`, `verify-email`, app-link, universal-link, or auth deep-link route files were found in the audited app tree.

## Risks Discovered From The Audit

1. Mobile password-reset entry is incomplete from a native UX perspective. A mobile user can request a reset link inside the app, but the generated link points to the web `/reset-password` route because the backend is web-link authoritative and the mobile app has no reset-token screen.

2. Mobile email-verification links are web-only. The Expo scheme exists, but neither backend email generation nor mobile deep-link routing currently maps verification links into native screens.

3. Environment drift can break links. Backend `WEB_APP_URL`, web `VITE_APP_URL`, and mobile `EXPO_PUBLIC_WEB_APP_URL` are separate variables with different responsibilities. If they point to different hosts, emails, web-generated public URLs, and mobile WebView routing can disagree.

4. The mobile `EXPO_PUBLIC_WEB_APP_URL` code fallback is `http://localhost:5173`, while current examples and backend local defaults use port `3000`. On a physical device, localhost is invalid unless explicitly replaced by a LAN IP.

5. Auth tokens are URL query params. Web reset and admin reset remove token params after success, and email-change removes its param after completion, but initial browser history, logs, analytics, screenshots, and email previews can still expose token-bearing URLs.

6. Email-change confirmation is routed through the general `/settings` route and processed inside `AccountSecuritySettings`. If this screen is reorganized later, the email-change token handler can silently break unless covered by route-level tests.

7. Notification email CTAs depend on backend target-url sanitization. The current standard is internal-path first, absolute conversion later. Allowing absolute external `targetUrl` values would create phishing/open-redirect risk.

8. Checkout callback behavior is intentionally backend-authoritative. Frontend code still passes a callback URL, which can mislead future engineers into thinking the client controls payment redirects.

9. Public profile and QR links correctly reject loopback hosts in non-local environments, but most other email links rely on the generic non-local `WEB_APP_URL` requirement. Production environment review remains mandatory before launch.

## Environment Routing Standard

Backend is the authoritative source for outbound email/auth absolute links.

Required backend standard:
- `WEB_APP_URL` must be set in every non-local environment.
- `WEB_APP_URL` must be the public browser web origin, with no trailing slash.
- `FRONTEND_URL` is legacy fallback only.
- `FRONTEND_PUBLIC_CHECKOUT_CALLBACK_URL` must be the full browser callback route for checkout, normally `<WEB_APP_URL>/bag/payment-return`.
- Production and staging must use HTTPS public domains, not localhost, LAN IPs, private IPs, or emulator hosts.
- Notification payload `targetUrl` must remain an internal path. Convert to absolute only when rendering email.

Required web standard:
- `VITE_API_BASE_URL` is the backend API origin.
- `VITE_APP_URL` is for web-generated public absolute URLs, not inbound auth-link resolution.
- Browser auth links must remain registered in `App.tsx` route configuration and covered by route-level tests when changed.
- Token-bearing auth pages should strip token params as soon as the token is no longer needed.

Required mobile standard:
- `EXPO_PUBLIC_API_BASE_URL` is the backend API origin reachable from the running device.
- `EXPO_PUBLIC_WEB_APP_URL` is the trusted web app origin used for Studio WebView and browser handoff, not currently a native auth-link target.
- Physical-device local testing must use a LAN IP, not `localhost`.
- If native auth links are later approved, mobile must add explicit reset/verify routes and route them from `app/_layout.tsx` / deep-link handling instead of overloading notification message routing.

Local development standard:
- Desktop browser email-link QA: backend `WEB_APP_URL=http://localhost:3000`.
- Physical mobile-device QA: backend `WEB_APP_URL=http://<LAN-IP>:3000`, mobile `EXPO_PUBLIC_WEB_APP_URL=http://<LAN-IP>:3000`, and frontend dev server bound to `0.0.0.0`.
- Emulator API QA may use emulator-specific API host handling, but email/browser URLs should still target a reachable web app origin.

## Phase 2 Backend Implementation Notes

Status: completed 2026-05-16.

Implemented backend-only hardening:
- Added `src/common/utils/auth-links.ts` as the focused auth email link builder on top of `resolveWebAppBaseUrl()`.
- Centralized construction for `/reset-password?token=...`, `/admin/reset-password?token=...`, `/verify-email?token=...&next=...`, and `/settings?tab=account-security&emailChangeToken=...`.
- Kept all current user-facing web routes stable. No mobile deep links, mobile reset-password route, frontend UI route change, Google auth, or Apple auth work was added.
- Encoded all auth-link token query values with `encodeURIComponent`.
- Sanitized verification `next` paths to same-app absolute paths only: single `/` prefix required, empty values rejected, protocol-relative `//...` rejected, and external `http://` / `https://` URLs rejected.
- Replaced backend manual auth-link string construction in `AuthService` and `EmailVerificationHelperService`.
- Preserved `WEB_APP_URL` as the primary backend frontend URL variable and `FRONTEND_URL` as the backward-compatible fallback.
- Hardened authenticated password changes so successful changes update the password, clear `mustResetPassword`, increment `authVersion`, and continue revoking other refresh tokens while preserving the current refresh session path.
- Added password-changed security alert emails after successful regular password reset and admin password reset, using `passwordChangedSecurityAlertEmail()` and the existing `auth.password.changed` scenario convention.
- Replaced the raw unknown/inactive password-reset email log with a short normalized email fingerprint via `maskEmailForLog()`.
- Added focused unit coverage for auth-link generation, safe `next` path filtering, email log fingerprinting, generic reset responses, WEB_APP_URL reset links, expired/reused reset-token rejection, refresh-token revocation, `authVersion` increments, and reset-success security alerts.

Route decisions intentionally deferred:
- Email-change confirmation remains on `/settings?tab=account-security&emailChangeToken=...` because Phase 1 confirmed the frontend handler exists there and no dedicated public email-change route exists.
- Password reset, email verification, admin reset, and email change remain web-link flows in Phase 2.
- Mobile native auth-link handling remains deferred until the product owner approves the routing model and mobile screens/routes.

Environment compatibility decisions:
- `WEB_APP_URL` remains authoritative for backend-generated auth/email links.
- `FRONTEND_URL` remains a fallback for older deployments.
- `WEB_APP_BASE_URL` was not introduced in Phase 2 to avoid creating a third backend naming path without a migration need.
- `.env.example` did not need further changes for Phase 2 because Phase 1 already documented the local `WEB_APP_URL=http://localhost:3000` standard.
